问卷系统设计框架与逻辑说明（全场景版）

一、系统定位

本系统不是传统意义上的“静态问卷发布工具”，而是一个“规则驱动的多角色问卷、评估、互评、投票与统计平台”。

传统问卷系统通常采用“固定题目 + 所有人作答同一内容 + 按题目汇总统计”的模式。这类系统适合满意度调查、信息收集、普通匿名问卷等基础场景，但对于以下复杂场景支持较弱：

1. 自评、互评、他评并存；
2. 不同身份回答不同问题；
3. 不同组、不同班级、不同部门看到不同对象或不同题目；
4. 评价对象不是固定写死，而是系统根据组织关系动态生成；
5. 投票对象、评分对象、候选人集合可按规则变化；
6. 统计维度不仅包括“题目”，还包括“人、组、角色、关系、轮次、权重、匿名策略”等。

因此，本系统应被定义为：

“一个以问卷模板为表现形式、以规则引擎为核心、以组织关系与评价关系为驱动、以多维统计与结果分析为目标的专业化评估与投票系统。”

二、系统目标

本系统的设计目标包括以下几个方面：

1. 支持基础问卷发布与回收；
2. 支持按身份、按组织、按条件的差异化题目展示；
3. 支持自评、互评、他评、评分、投票、排序、矩阵评价等复杂业务；
4. 支持“谁评价谁”的动态关系生成；
5. 支持匿名、实名、半匿名等不同作答模式；
6. 支持专业的数据统计、交叉分析、加权计算与结果导出；
7. 支持本地 Python 管理内核运行，并最终打包为 exe，脱离 Python 环境独立使用；
8. 支持后续扩展为局域网版、校园内部署版或云端版。

三、系统设计总原则

1. 配置优先，而非硬编码优先
系统中的问卷、题目、展示条件、评价对象、统计规则应尽可能通过配置描述，而不是写死在代码中。

2. 问卷只是表现形式，规则才是系统核心
系统表面上是“填写网页”，实质上是根据业务规则生成页面、控制权限、组织关系和统计逻辑。

3. 参与者与账号分离
“登录账号”与“业务参与者”必须分离。一个账号对应一个使用者，但一个使用者在业务中可能既是答卷人，也是被评价对象。

4. 作答人与评价对象分离
谁在填写、他在评价谁、他以何种身份填写，必须分别建模。

5. 结果统计不只按题，还要按对象、关系和上下文统计
系统必须天然支持多维分析，而不是只做题目频数统计。

6. 前后端职责明确
HTML 页面负责渲染与交互，Python 后端负责规则计算、页面生成、权限校验、数据处理与统计。

四、核心抽象模型

为了覆盖全场景，本系统需要建立以下核心抽象。

（一）模板（Template）
模板用于定义问卷内容本身，包括：
1. 问卷标题、说明；
2. 页面结构；
3. 题块与题目；
4. 题型定义；
5. 校验规则；
6. 默认显示逻辑；
7. 可重复渲染的题组结构。

模板回答的是：“这个业务可能会问哪些问题、以什么形式问。”

（二）任务（Task / Campaign）
任务是模板在某个时间段、某个组织环境下的一次实际发布。
例如：
1. 2026 年班委互评；
2. 党员民主评议；
3. 项目答辩评委评分；
4. 奖学金候选人投票；
5. 新生满意度调查。

任务回答的是：“这一次具体由谁参与、何时参与、在什么规则下参与。”

（三）参与者（Participant）
参与者是业务对象，不等于登录账号。参与者可具有多种属性：
1. 姓名；
2. 学号或工号；
3. 班级；
4. 部门；
5. 小组；
6. 身份标签；
7. 是否候选人；
8. 是否可被评价；
9. 是否可投票；
10. 其他扩展字段。

（四）角色（Role）
角色用于决定权限和题目展示逻辑。常见角色包括：
1. 普通成员；
2. 班委；
3. 学生；
4. 教师；
5. 评委；
6. 候选人；
7. 管理员；
8. 超级管理员。
这些只是具体情况中的角色，实际应按照权限划分，而不是具体角色分类。

同一个参与者在不同任务中可以具有不同角色。

（五）目标对象（Target）
在复杂问卷中，很多题目不是单纯“填写意见”，而是“对某个对象进行评价或投票”。
因此系统必须引入目标对象概念。目标对象可能是：
1. 自己；
2. 同组成员；
3. 班委成员；
4. 候选人；
5. 项目作品；
6. 某个部门；
7. 某个活动方案；
8. 某个课程或教师。

（六）关系（Relation）
关系是系统最关键的业务抽象之一。它回答的问题是：
“谁，以什么身份，对谁，执行什么类型的作答或评价？”

例如：
1. 张三 -> 张三，自评；
2. 张三 -> 李四，互评；
3. 教师甲 -> 学生乙，教师评价；
4. 评委甲 -> 项目 A，评审打分；
5. 党员甲 -> 支委成员乙，民主评议；
6. 学生甲 -> 候选人丙，投票。

没有关系模型，就无法真正专业地支持复杂业务。

（七）作答会话（Response Session）
每一次完整作答都应视为一个会话。会话记录以下信息：
1. 属于哪个任务；
2. 谁在填写；
3. 评价对象是谁；
4. 当前关系类型是什么；
5. 作答起止时间；
6. 当前状态（未开始、进行中、已提交、已撤回等）；
7. 是否匿名；
8. 提交凭证。

会话是答案的容器。

（八）答案（Answer）
答案记录具体题目的作答内容，包括：
1. 对应题目；
2. 对应作答会话；
3. 选项值；
4. 文本内容；
5. 数值分；
6. 排序结果；
7. 矩阵项答案；
8. 附件信息（如需支持）。

五、全场景业务覆盖能力

本系统必须覆盖至少以下场景。

（一）普通统一问卷
定义：所有人回答相同问题。
适用场景：满意度调查、信息收集、活动反馈、报名表。
系统需求：
1. 支持常见题型；
2. 支持必填校验；
3. 支持提交与导出；
4. 支持基础统计。

（二）条件问卷
定义：不同身份、不同条件下看到不同题目或题块。
适用场景：分角色调查、流程分支问卷、预筛选问卷。
系统需求：
1. 支持基于身份显示；
2. 支持基于组织属性显示；
3. 支持基于前序答案显示；
4. 支持题块级条件控制；
5. 支持动态必填逻辑。

（三）自评问卷
定义：每个参与者仅对自己作答。
适用场景：个人总结、自我评价、自我诊断、自我申报。
系统需求：
1. 系统自动将目标对象绑定为当前作答人；
2. 自评内容可与其他关系类型分离统计；
3. 可单独设置自评权重。

（四）互评问卷
定义：参与者对指定范围内的其他参与者进行评价。
适用场景：组内互评、班委互评、成员协作评价、奖学金互评。
系统需求：
1. 系统按规则生成可评价对象集合；
2. 支持排除自己；
3. 支持仅评价同组成员；
4. 支持评价指定数量对象；
5. 支持一人对多人连续评价；
6. 支持按对象统计结果。

（五）他评问卷
定义：某一类角色评价另一类角色。
适用场景：教师评价学生、学生评价教师、上级评价下级、评委评价选手。
系统需求：
1. 支持跨角色关系；
2. 支持目标集规则生成；
3. 支持不同关系下使用不同模板或题块；
4. 支持不同身份视图。

（六）投票系统
定义：参与者对候选对象进行选择、排序或打分。
适用场景：评优评先、班委竞选、作品投票、民主推荐。
系统需求：
1. 支持单选投票；
2. 支持多选投票；
3. 支持限选数量；
4. 支持排序投票；
5. 支持评分投票；
6. 支持候选人列表动态生成；
7. 支持防重复、防刷票；
8. 支持实名或匿名投票。

（七）评分系统
定义：参与者根据评价指标对目标对象进行打分。
适用场景：项目评审、答辩评分、绩效考核、综合测评。
系统需求：
1. 支持分项评分；
2. 支持权重计算；
3. 支持总分自动汇总；
4. 支持评委独立打分；
5. 支持去极值、求均值等统计策略；
6. 支持按项目、按评委、按组别输出结果。

（八）分组问卷
定义：不同组、班级、部门、寝室看到不同题目或不同对象。
适用场景：班级管理、部门评审、宿舍评比、项目分组反馈。
系统需求：
1. 支持组织结构导入；
2. 支持同组范围规则；
3. 支持跨组规则；
4. 支持组内和组间分别统计。

（九）多轮次问卷或评审
定义：同一业务分多个阶段开展，不同轮次题目或规则不同。
适用场景：初评、复评、终评，多轮筛选。
系统需求：
1. 支持轮次定义；
2. 支持轮次间数据隔离；
3. 支持轮次间晋级条件；
4. 支持轮次历史结果追踪。

（十）匿名、实名与半匿名场景
定义：系统对作答身份暴露程度可配置。
系统需求：
1. 实名模式：管理员与统计层可看到作答人与答案绑定关系；
2. 匿名模式：只记录已提交状态，不暴露具体作答内容来源；
3. 半匿名模式：管理层可追溯，统计报表中隐藏身份；
4. 支持匿名 token 与防重复机制。

六、系统逻辑主线

系统运行逻辑应遵循以下主线：

第一步：定义组织与参与者
管理员导入或维护参与者信息、组别信息、角色信息、组织结构信息。

第二步：定义模板
管理员创建模板，配置题块、题目、选项、题型、校验规则、说明文本。

第三步：定义任务
管理员基于模板发布任务，设置时间范围、作答对象、匿名策略、提交限制等。

第四步：定义规则
管理员为任务配置：
1. 谁能填写；
2. 谁能评价谁；
3. 谁能看到哪些题；
4. 某些题是否只在特定条件下显示；
5. 最终如何统计和加权。

第五步：系统生成作答关系
系统根据参与者、组织结构和规则，自动生成“作答人 - 目标对象 - 关系类型”的可作答集合。

第六步：前端按当前上下文渲染页面
当某个用户进入问卷时，系统根据当前身份、当前任务、当前目标对象、显示规则和模板结构，动态生成页面 schema，并由 HTML 页面渲染。

第七步：提交与校验
系统对答案进行：
1. 必填校验；
2. 类型校验；
3. 逻辑校验；
4. 重复提交校验；
5. 权限校验；
6. 时间窗口校验。

第八步：结果入库
系统将答案写入作答会话，并形成可供统计的标准化数据。

第九步：统计分析
系统按题目、人员、组别、角色、关系类型、轮次、权重等多维度生成统计结果。

第十步：报表与导出
系统支持导出 Excel、CSV、PDF 等结果，用于归档、展示、评审或后续分析。

七、规则引擎设计

规则引擎是本系统的核心，应至少包括以下几类规则。

（一）准入规则
定义谁能参与某任务。
例如：
1. 仅 2024 级学生可参加；
2. 仅班委可填写干部自评；
3. 仅评委账号可进行项目评分。

（二）目标生成规则
定义某个作答人可以评价哪些目标对象。
例如：
1. 每个人只能评价自己；
2. 每个人评价同组成员，排除自己；
3. 教师评价自己授课班级学生；
4. 评委评价所有候选项目；
5. 党员评价支委全体成员。

（三）显示规则
定义哪些题或题块在什么条件下显示。
例如：
1. 若角色为班长，则显示班级管理题块；
2. 若上一题选择“参加过活动”，则显示活动体验部分；
3. 若目标对象是干部，则显示履职评价部分；
4. 若为匿名模式，则隐藏某些说明信息。

（四）校验规则
定义提交时的逻辑约束。
例如：
1. 投票最多选 3 人；
2. 必须完成全部目标对象评价后才能提交；
3. 分数必须在 1 到 10 之间；
4. 排序题不得重复排序。

（五）去重与限制规则
定义重复作答与次数限制。
例如：
1. 每个账号仅可提交一次；
2. 每个目标对象仅可评价一次；
3. 一个投票任务只允许一次最终提交；
4. 可保存草稿但不可重复正式提交。

（六）统计规则
定义结果如何汇总。
例如：
1. 自评占 20%，互评占 30%，教师评分占 50%；
2. 互评时去掉最高分与最低分；
3. 对文本题进行词频分析；
4. 对矩阵题按维度输出均值。

八、页面渲染逻辑

前端 HTML 页面不应写死具体题目，而应作为“动态表单渲染器”。

后端应根据当前会话生成页面描述数据，包含：
1. 当前任务信息；
2. 当前作答人信息；
3. 当前目标对象信息；
4. 当前应显示的题块与题目；
5. 各题型的配置参数；
6. 选项列表；
7. 必填规则；
8. 条件显示结果；
9. 提交限制说明。

前端职责：
1. 渲染页面；
2. 响应交互；
3. 执行基础前端校验；
4. 收集答案；
5. 调用提交接口。

后端职责：
1. 计算页面内容；
2. 核验当前身份与权限；
3. 处理业务规则；
4. 保存答案；
5. 执行最终校验；
6. 生成统计结果。

这种设计可保证后续系统扩展时，不必频繁修改前端页面结构。

九、数据模型逻辑

从逻辑上看，系统最核心的数据链为：

用户账号 -> 参与者 -> 所属组织/角色 -> 所属任务 -> 评价关系 -> 作答会话 -> 答案 -> 统计结果

关键思想如下：

1. 用户账号用于认证；
2. 参与者用于业务建模；
3. 组织与角色用于规则计算；
4. 任务用于实际发布；
5. 评价关系用于生成“谁评谁”；
6. 作答会话用于承载具体一次填写；
7. 答案用于保存每道题的数据；
8. 统计结果用于输出分析视图。

十、统计与分析框架

专业问卷系统的统计不应只停留在“第 1 题选 A 的有多少人”这一层，而应支持以下维度：

（一）按题统计
1. 单选题各选项频数；
2. 多选题各选项覆盖率；
3. 评分题均值、中位数、方差；
4. 文本题词频、关键词；
5. 矩阵题按行列统计。

（二）按人统计
1. 某人收到的平均评分；
2. 某人被多少人评价；
3. 某人不同维度的得分画像；
4. 某人历次轮次结果变化。

（三）按组统计
1. 各组平均分；
2. 各组完成率；
3. 各组内部互评差异；
4. 组间比较。

（四）按角色统计
1. 学生评价教师结果；
2. 教师评价学生结果；
3. 班委与普通成员结果对比；
4. 不同评委群体结果差异。

（五）按关系类型统计
1. 自评结果；
2. 互评结果；
3. 他评结果；
4. 投票结果；
5. 多来源加权结果。

（六）按轮次统计
1. 初评、复评、终评分别统计；
2. 多轮结果对比；
3. 晋级与淘汰依据展示。

（七）综合加权统计
1. 根据预定义权重进行综合评分；
2. 支持公式化汇总；
3. 支持不同任务间指标映射；
4. 支持结果快照保存，确保历史可追溯。

十一、匿名与安全逻辑

为了使系统在专业场景下可用，匿名和安全机制必须单独设计。

（一）匿名模式设计
1. 系统记录“已完成”状态，但作答内容与身份脱钩；
2. 系统可通过匿名 token 控制一次性提交；
3. 统计结果中不显示具体身份。

（二）半匿名模式设计
1. 管理员可确认谁提交了；
2. 统计分析环节默认隐藏作答人；

（三）实名模式设计
1. 作答人与答案直接绑定；
2. 适用于工作考核、责任申报、审批类场景。

十三、典型业务流程示例

以下给出几类典型场景的逻辑链。

（一）自评场景
1. 导入参与者；
2. 定义自评模板；
3. 发布任务；
4. 系统为每人生成一个“自己 -> 自己”的作答关系；
5. 用户登录后只看到自己的自评页面；
6. 提交后按人统计自评分。

（二）组内互评场景
1. 导入分组信息；
2. 定义互评模板；
3. 配置规则：每人评价同组其他成员，不含自己；
4. 系统自动计算每人对应的目标集合；
5. 页面按目标对象动态展开评价题组；
6. 最终按被评价对象统计组内平均分。

（三）评委评分场景
1. 导入评委与候选项目；
2. 配置规则：每位评委对全部项目评分；
3. 使用评分模板；
4. 系统为每个“评委 -> 项目”生成评分会话；
5. 统计时按项目汇总，并支持权重、去极值等算法。

（四）匿名投票场景
1. 导入投票人和候选人；
2. 设置匿名模式；
3. 设置限选数量；
4. 用户通过唯一身份进入投票页面；
5. 系统保证一次性提交；
6. 导出时只保留投票结果，不暴露投票人具体选择。

十四、技术实现建议

基于你的目标，即“管理员软件为本地 Python 内核，问卷为 HTML 网页形式，最终打包为 exe 且不依赖 Python”，推荐采用如下技术路线：

1. 后端：Python（推荐 FastAPI 或 Flask）；
2. 前端：HTML + CSS + JavaScript；
3. 数据库：SQLite 作为初期单机方案；
4. 管理端：本地 Python 服务 + 浏览器访问本地页面，或桌面壳嵌入网页；
5. 打包方式：PyInstaller 打包为 exe；
6. 导入导出：支持 Excel/CSV；
7. 图表：前端图表库或后端导出统计结果。

这样做的优势在于：
1. HTML 页面表现力强；
2. Python 后端便于规则处理；
3. SQLite 适合单机部署；
4. 打包后可脱离 Python 环境运行；
5. 后续易于扩展为局域网部署或服务器部署。

十六、最终结论

本问卷系统的本质，不应被理解为“发题和收题的网页程序”，而应被理解为：

“一个以规则引擎为核心，以参与者、角色、组织、目标对象、评价关系和统计模型为基础的数据采集与评估平台。”

只有建立这一层抽象，系统才能真正覆盖全场景，包括：
1. 普通问卷；
2. 条件问卷；
3. 自评；
4. 互评；
5. 他评；
6. 投票；
7. 评分；
8. 分组问卷；
9. 多轮评审；
10. 匿名与实名混合场景；
11. 多维统计与综合加权分析。

因此，系统设计的关键不是“题目怎么显示”，而是以下四个核心问题：

1. 谁可以进入系统；
2. 谁在填写；
3. 他在评价谁；
4. 最终如何统计与解释结果。

只要这四个问题的框架设计正确，本系统就具备了专业化、可扩展、可复用的底层能力，并能够支撑后续实际开发、项目展示、竞赛答辩和长期演进。
